# Epic 21: Multi-Version Summaries (per model)

**Phase**: 2 — Summarization Quality  
**Status**: 🔵 Planned  
**Priority**: 🟡 P2

## Goal

Store one summary per `(video, model)` combination instead of one summary per video.
Allows comparing results from different models without losing previous work.

---

## User Stories

---

### US-2101: Summaries stored per model

**As a** user experimenting with different Ollama models  
**I want** each model's summary to be saved separately  
**So that** I don't lose the previous result when I switch models and re-run

#### Acceptance Criteria

**Given** a video with an existing summary generated by `qwen3:8b`  
**When** I select `qwen2.5-coder:14b` and click "Summarize"  
**Then** a new summary is generated and saved for `qwen2.5-coder:14b`, and the `qwen3:8b` summary remains unchanged

**Given** a video with an existing summary generated by `qwen3:8b`  
**When** I select `qwen3:8b` again and click "Re-run summary"  
**Then** the `qwen3:8b` summary is overwritten with the new result

**Given** a video with summaries from two different models  
**When** I open the result page  
**Then** I see the most recently generated summary by default

#### Edge Cases

- First summarization for a video — behaves as before, no UI change
- Model field is null — falls back to current single-summary behaviour
- Summary generation fails — existing entry for that model is not overwritten

#### Out of Scope

- Diffing or side-by-side comparison view (see US-2102)
- Summaries from different prompt versions of the same model
- Automatic regeneration when model changes

#### Notes for Engineering

- New table `summaries` with columns: `id`, `video_id` (FK → videos.id), `model`, `mode`, `chunks_count`, `summary_text`, `duration_seconds`, `created_at`
- Unique constraint on `(video_id, model)` — upsert on re-run
- `GET /api/result/{video_id}` returns all summaries for the video as a list
- Active summary = most recently created/updated entry
- Existing `summary_*` columns on `subtitles_formatted` become deprecated once migrated
- Migration: copy existing `summary_text` + `summary_model` to new table on first run

---

### US-2102: View and switch between saved versions

**As a** user who has run summarization with multiple models  
**I want** to see all saved versions and switch between them  
**So that** I can compare outputs without re-generating anything

#### Acceptance Criteria

**Given** a video with summaries from 2+ models  
**When** I open the Summary tab  
**Then** I see a list of saved versions (model name + date) and can click any to view it

**Given** I am viewing a summary version that is not the most recent  
**When** I look at the UI  
**Then** it is clearly indicated which version I am viewing

**Given** a video with only one saved summary  
**When** I open the Summary tab  
**Then** no version switcher is shown — behaviour is identical to today

#### Edge Cases

- Deleting a version: allowed, with confirmation. If it was the active version, switch to the most recent remaining one
- All versions deleted: Summary tab shows "No summary yet" state

#### Out of Scope

- Side-by-side diff view
- Exporting comparison results
- Sorting versions by quality score

#### Notes for Engineering

- Version switcher UI: compact list of model name badges above the summary text, similar to git branch selector style. Only shown when 2+ versions exist.
- DELETE endpoint: `DELETE /api/result/{video_id}/summary/{summary_id}`
- Frontend stores `activeSummaryId` in local state (not persisted — resets to latest on page reload)
